# EntityReskin Manifest Format (MANIFEST)

The contract between the resource backend and the client. The hosted backend (the `web` module) generates a manifest per server dynamically and issues time-limited download URLs; a self-hoster can also produce a manifest in the same format from a static directory with ordinary relative paths, and the client consumes both forms identically.

The implementation lives in the `shared` module under `io.github.zzz1999.entityreskin.protocol.manifest.*`, with parsing and validation entered through `Manifests.parse(String)`; the hosted backend's generator is `ManifestService` in the `web` module.

## Fetching and trust

1. At handshake the server sends, via `SET_MANIFEST_SOURCE`, the `baseUrl` + `manifestPath` + the **manifest's SHA-256** (which the plugin polled from the backend — see PROTOCOL.md, "Plugin ↔ resource backend").
2. The client downloads `baseUrl + manifestPath` over HTTPS, first verifying that the overall SHA-256 matches the sent value (**hash pinning / MITM protection**), then parses it.
3. Each resource is downloaded from `baseUrl + path`, **computing SHA-256 as it streams** and comparing against the manifest; a mismatch discards it.
4. Resource updates: backend content changes (or the signing window flips) → the manifest hash changes → the plugin discovers it on the next poll and re-sends `SET_MANIFEST_SOURCE`, and the client re-fetches.

## Signed download URLs and the rolling window (hosted backend)

- A `path` generated by the hosted backend looks like: `download/<sha256>?exp=<expiry seconds>&srv=<server token>&sig=<HMAC-SHA256 lowercase hex>`. The signature binds the (blob hash, expiry, server token) triple; tampering with any one invalidates it. Signatures are always lowercase hex and introduce no characters that would interfere with URL semantics.
- **Rolling window**: `window = floor(epochSeconds / (TTL/2))`, `exp = (window + 2) × (TTL/2)`. A URL is valid for between TTL/2 and TTL (default TTL 24h); within one half-window the manifest is **byte-for-byte stable** (entries sorted by identifier, no timestamp fields), so the hash can be pinned.
- When a URL expires (a very long session) the client simply re-fetches the manifest to obtain fresh signatures.
- Download responses are served as `application/octet-stream` with `Content-Disposition: attachment` (never inline) and are rate-limited by the server's token-bucket byte rate.

## JSON structure

```json
{
  "formatVersion": 1,
  "baseUrl": "https://cdn.example.com/entityreskin/",
  "entries": {
    "entityreskin:dragon_red": {
      "displayName": "Red Dragon",
      "geometryName": "geometry.dragon_red",
      "defaultAnimation": "animation.dragon_red.idle",
      "renderControllerEntry": "controller.render.dragon_red",
      "resources": {
        "geometry":  { "path": "blobs/ab/<sha256>.geo",  "sha256": "<64-char lowercase hex>", "size": 184320 },
        "animation": { "path": "blobs/cd/<sha256>.anim", "sha256": "<64-char lowercase hex>", "size": 65536 },
        "texture":   { "path": "blobs/ef/<sha256>.png",  "sha256": "<64-char lowercase hex>", "size": 262144 },
        "animation_controllers": { "path": "...", "sha256": "...", "size": 8192 },
        "render_controllers":    { "path": "...", "sha256": "...", "size": 4096 }
      }
    }
  }
}
```

## Field reference

| Field | Description |
|---|---|
| `formatVersion` | Must currently be `1`, otherwise rejected |
| `baseUrl` | Prefix for all relative `path`s; must be HTTPS (only `localhost`/`127.0.0.1` may use HTTP, for local development), enforced by `Manifests.validate` |
| `entries` | `identifier → entry`, non-empty |
| `displayName` | Display name (optional) |
| `geometryName` | Geometry name (Bedrock `geometry.*`) |
| `defaultAnimation` | Animation played by default in the MVP (used until animation controllers are ready) |
| `renderControllerEntry` | Render controller entry (phase 3, optional) |
| `resources` | `kind → file`, non-empty |

**Resource kinds**: `geometry`, `animation`, `texture` (required in the MVP); `animation_controllers` (phase 2) and `render_controllers` (phase 3) are optional.

**Each file**: `path` (relative, safe), `sha256` (64-char lowercase hex), `size` (bytes, ≥ 0, used to reject oversized files before download).

## Content-addressing convention

A blob's file name is its SHA-256 (e.g. `blobs/<first 2 hex>/<full hash>`). Benefits: unchanged content → unchanged URL → natural caching; changed content → changed hash → automatic invalidation; the file name is the checksum → tamper-evident. The client's local cache is likewise content-addressed as `cache/<first 2>/<sha256>`.

## Validation rules (`Manifests.validate`, a security boundary)

- `formatVersion == 1`; `entries` non-empty; every entry's `resources` non-empty.
- Each file: `sha256` is 64-char lowercase hex; `size >= 0`; `path` is a **safe relative path**.
- **Safe relative path**: non-empty, no backslash `\`, does not start with `/`, contains no `:` (excludes drive letters / protocols), contains no `..` segment. This prevents path traversal.
